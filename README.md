# AI Evaluation & Benchmarking Platform (LLM Observability & Cost Analysis)

## Architecture Overview

Dataset → Evaluation Engine → Model API → LLM Judge → Run Tracing → Dashboard

The platform consists of four core components:

1. **Evaluation Engine (Backend)**  
   Executes model prompts across supported providers via OpenRouter and records responses.

2. **LLM Judge Pipeline**  
   Uses a structured evaluation prompt to score model outputs for:
   - accuracy
   - hallucination detection
   - reasoning quality
   - relevance

3. **Run Tracing System**  
   Captures full telemetry for every prompt-response interaction including:
   - raw model completion
   - judge prompt
   - judge JSON output
   - latency
   - token usage

4. **Dashboard Interface**  
   Displays experiment runs, traces, and aggregated metrics while safely ignoring failed evaluations.

## Platform Features

### 1. Dataset Management & Upload
You can easily create your own evaluation benchmarks. The platform supports:
- **Manual entry** of prompts and expected outputs via the web UI.
- **Bulk uploads** using CSV or JSON files.
  - Uploaded files must conform to a basic schema, containing at least `prompt` and `expected_output` (or `expected`) fields.
  - Optional metadata like `evaluation_type` and `difficulty` are also supported.

### 2. Run Tracing (Observability)
An observability pipeline inspired by LangSmith-style run tracing is built directly into the platform. When an evaluation run executes, the backend captures detailed telemetry for every single prompt/response interaction and persists it in the SQLite database automatically.

The **Run Tracing** dashboard allows you to inspect:
- **Exact Evaluation Inputs:** The exact inputs and expected outputs sent to the model.
- **Model Responses:** The raw completion generated by the AI model.
- **Judge Context:** The exact prompt synthesized for the OpenRouter Judge LLM.
- **Raw Judge Telemetry:** The unparsed JSON response direct from the LLM evaluator, allowing you to debug hallucination triggers and reasoning quality scoring.
- **Latency & Tokens:** Response generation time and real token usage measured across the evaluation.

### 3. Model Comparison Dashboard
Design experiments to compare different providers side-by-side using the **Experiments Dashboard**. 
- Aggregate and visualize **Accuracy**, **Relevance**, and **Hallucination** rates using bar charts and multi-dimensional radar plots.
- The details table splits out performance per metric while safely isolating and excluding **Failed Runs/Timeouts** from dragging down aggregate data.

### 4. Real Cost Tracking
Instead of relying on estimations, the platform computes token costs by multiplying the real prompt & completion token consumption against a precise per-1M token pricing configuration. 
- *Note: Cost is only shown when real usage metadata and pricing data are available in the system logic. If pricing is unavailable for a model, the metric safely falls back to displaying N/A.*

## Judge Pipeline & Resilience

The platform uses an **LLM-as-a-judge** pattern (via OpenRouter) to evaluate model outputs. To handle upstream instability (rate limits, context windows, 502s), the judge pipeline includes several resilience mechanisms:

1. **Retry Behavior**: The primary LLM judge will retry failures up to 3 times with an exponential backoff (2s, 4s, 6s) before conceding.
2. **Fallback Judge**: If the OpenRouter API remains unreachable after all retries, the system transparently shifts to a local heuristic Fallback Judge. 
    * *Note: The Fallback Judge is purely a resilience mechanism (using token containment, fuzzy stem matching, and regex-based hallucination detection) to prevent pipeline crashes. It is not intended to replace the primary LLM judge.*
3. **Failed-Run Handling**: If an evaluated model itself fails to generate a response (e.g. returning API limits), the run is marked as `failed` and metrics are safely ignored rather than skewing the dashboard aggregates.

## Quick Start

1. **Clone the repository**
```bash
git clone https://github.com/iqbalxo/AI-Evaluation-And-Benchmarking.git
```

2. **Install backend dependencies**
```bash
cd backend
pip install -r requirements.txt
```

3. **Set environment variable**
```bash
# On Linux/macOS
export OPENROUTER_API_KEY="your_key_here"

# On Windows PowerShell
$env:OPENROUTER_API_KEY="your_key_here"
```

4. **Start the backend server**
```bash
uvicorn main:app --reload --port 8000
```

5. **Start the frontend app** (in a new terminal)
```bash
cd frontend
npm install
npm run dev
```

## Validation Suite

You can verify the accuracy of both the primary LLM judge and the fallback heuristic by running the judge validation suite (which guarantees 10/10 passes on ground-truth assertions):

```bash
cd backend
# Run the live LLM judge suite (requires OPENROUTER_API_KEY)
python test_judge.py

# Run the isolated fallback judge suite
python test_judge.py --fallback-only
```
