# AI Evaluation & Benchmarking Platform

## Run Tracing (Observability)
A LangSmith-style observability pipeline is built directly into the platform. When an evaluation run executes, the backend captures detailed telemetry for every single prompt/response interaction and persists it in the SQLite database automatically.

## Architecture Overview

The platform consists of four core components:

1. **Evaluation Engine (Backend)**  
   Executes model prompts across supported providers via OpenRouter and records responses.

2. **LLM Judge Pipeline**  
   Uses a structured evaluation prompt to score model outputs for:
   - accuracy
   - hallucination detection
   - reasoning quality
   - relevance

3. **Run Tracing System**  
   Captures full telemetry for every prompt-response interaction including:
   - raw model completion
   - judge prompt
   - judge JSON output
   - latency
   - token usage

4. **Dashboard Interface**  
   Displays experiment runs, traces, and aggregated metrics while safely ignoring failed evaluations.

The **Run Tracing** dashboard allows you to inspect:
- **Exact Evaluation Inputs:** The exact inputs and expected outputs sent to the model.
- **Model Responses:** The raw completion generated by the AI model.
- **Judge Context:** The exact prompt synthesized for the OpenRouter Judge LLM.
- **Raw Judge Telemetry:** The unparsed JSON response direct from the LLM evaluator, allowing you to debug hallucination triggers and reasoning quality scoring.
- **Latency & Tokens:** Response generation time and approximate token usage across the evaluation.

*Access the traces by navigating to the "Run Tracing" (🔍) sidebar menu in the web UI.*

## Judge Pipeline & Resilience

The platform uses an **LLM-as-a-judge** pattern (via OpenRouter) to evaluate model outputs. To handle upstream instability (rate limits, context windows, 502s), the judge pipeline includes several resilience mechanisms:

1. **Retry Behavior**: The primary LLM judge will retry failures up to 3 times with an exponential backoff (2s, 4s, 6s) before conceding.
2. **Fallback Judge**: If the OpenRouter API remains unreachable after all retries, the system transparently shifts to a local heuristic Fallback Judge. 
    * *Note: The Fallback Judge is purely a resilience mechanism (using token containment, fuzzy stem matching, and regex-based hallucination detection) to prevent pipeline crashes. It is not intended to replace the primary LLM judge.*
3. **Failed-Run Handling**: If an evaluated model itself fails to generate a response (e.g. returning API limits), the run is marked as `failed` and metrics are safely ignored rather than skewing the dashboard aggregates.

### Validation Suite

You can verify the accuracy of both the primary LLM judge and the fallback heuristic by running the judge validation suite (which guarantees 10/10 passes on ground-truth assertions):

```bash
cd backend
# Run the live LLM judge suite (requires OPENROUTER_API_KEY)
python test_judge.py

# Run the isolated fallback judge suite
python test_judge.py --fallback-only
```
